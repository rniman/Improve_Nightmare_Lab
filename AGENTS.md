# AGENTS.md

## Project Context

This repository contains a legacy graduation project —
a multiplayer game built on a custom framework in C/C++.

**Tech stack:**
- Language: C/C++ (C++20, MSVC v143)
- Graphics: DirectX 12, HLSL shaders
- Audio: FMOD
- Platform: Win32 API
- Networking: TCP (WsaAsyncSelect)

The current focus is **incremental refactoring** to improve
readability, maintainability, and system boundaries.
Large architectural rewrites should be avoided unless explicitly requested.

## Do Not Modify

The following files are **external dependencies or generated code**.
Never modify them unless explicitly instructed:

- `d3dx12.h` — DirectX 12 helper header (Microsoft)
- `DDSTextureLoader12.h / .cpp` — DDS texture loader (Microsoft)
- `targetver.h`
- `Resource.h` — auto-generated by Visual Studio resource editor

**Minimize changes to:**
- `stdafx.h` / `stdafx.cpp` — precompiled header.
  Only add includes here when genuinely needed project-wide.

## Refactoring Policy

1. Prefer **small incremental changes**.
2. **Preserve existing behavior** unless explicitly instructed otherwise.
3. Avoid rewriting entire systems at once.
4. When architecture is unclear, **analyze the code before modifying it**.
5. Favor improving readability and responsibility separation
   over introducing new abstractions.

---

## Code Analysis First

Before modifying code:

1. Explain the current implementation.
2. Identify problems or maintainability issues.
3. Propose the smallest possible refactor.

If something is unclear in the architecture, state assumptions explicitly.
Do not implement changes until the refactoring plan is approved.

## Rendering System Notes

The rendering system is DirectX 12-based with the following frame flow:

```
GameFramework (main loop)
  → Scene::Update (game logic, animation)
  → Scene::Render (submit draw calls per Shader/Object)
  → Shader sets PSO and root signature
  → Object records draw commands to command list
  → GPU executes → Present
```

If refactoring renderer code:
- Do not change rendering behavior unless requested.
- Focus on improving responsibility separation.
- Avoid restructuring the entire rendering pipeline.
- Be careful with root signature and descriptor heap changes.

For detailed architecture analysis, see `docs/architecture.md` (if available).

---

## Resource Management Notes

Resource ownership and lifetime should remain explicit.

When modifying resource-related code:
- Avoid introducing hidden global dependencies.
- Prefer clear ownership and lifetime management.
- Avoid implicit coupling between renderer and gameplay systems.
- Be cautious with COM object (`ComPtr`) lifetimes and GPU resource barriers.

---

## Modification Scope Rules

When performing changes:
- Modify **only the relevant module**.
- Avoid cross-system refactors unless explicitly requested.
- Do not rename or move large parts of the project without explanation.
- Do not reorganize directory structure as a side effect of other work.
- Avoid introducing new third-party libraries.
- Avoid changing public APIs unless required for the refactor.
